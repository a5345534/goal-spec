OpenSpec decision-review explainer

clarify-value-gated-writer-workflow

此 change 將 agent-goal-writer 明確定義為「會挑戰價值的規格夥伴」：先與使用者釐清與辯論，再經過 pre-spec 品質閘門，最後才寫 OpenSpec。

Why What Changes Before/After Flow 決策點 實作切片 驗證 風險 Agent prompts

為什麼要做

目前 skill 已經說明 discovery、spec kernel、OpenSpec 寫作與驗證，但仍可能被理解為「使用者說什麼就產生規格」。這會讓模糊或價值不足的目標被包裝成看似成熟的 proposal/design/tasks/spec。

此 change 的核心價值是：在寫 spec 前，讓 writer 先挑戰目標是否值得做、成功訊號是否可驗證、是否有 no-build 或更小範圍的替代方案。

決策：在 agent-goal-writer 中加入 value-gated workflow。OpenSpec 寫作必須先通過 pre-spec gate，或在使用者明確承認風險後以 proceed_with_assumptions 前進。

變更內容

Critical collaborator

writer 不再只是 order-taker；必須能提出建設性反對與替代方案。

Value Challenge Gate

在 spec kernel 前檢查問題、受影響者、價值證據、不做的代價、no-build、smaller-scope。

Workflow script

新增 helper contract，建立 .writer-workflow artifacts 並在寫 OpenSpec 前執行品質閘門。

變更前 / 變更後

變更前 Before

  • 模糊需求可能太快被轉成 OpenSpec。
  • 價值挑戰存在於 prompt 風格，但不是明確 gate。
  • 沒有 workflow artifact 可供審查 pre-spec 品質。

變更後 After

  • writer 必須先釐清問題與挑戰價值。
  • pre-spec gate 產生 blocked/pass/proceed_with_assumptions。
  • proposal/design 必須保留價值辯論、替代方案、假設與風險。

決策點 Decision Points

決策 1：使用 state-machine workflow

推薦：採用 INTAKE → VALUE_CHALLENGE → CLARIFICATION → SPEC_KERNEL → PRE_SPEC_QUALITY_GATE → OPENSPEC_WRITE → VALIDATION。

取捨：比純 prompt 多一些 artifacts，但可審查、可重跑、可阻擋 premature spec。

信心：高。此設計直接回應「每個階段穩定運行」的需求。

決策 2：使用 blocked/pass/proceed_with_assumptions

推薦：允許帶假設前進，但必須明確記錄使用者承認的風險。

取捨：避免無限阻擋，也避免把未驗證風險偽裝成事實。

信心：中高。需透過測試防止此狀態被濫用。

Implementation slices 實作切片

Slice 1：Skill contract

更新 SKILL.md 的目的、workflow、品質 gate 與 guardrails。驗收：文件明確禁止過早寫 spec。

Slice 2：Workflow script

新增 init/check/gate/write-spec CLI。驗收：blocked gate 時 write-spec 失敗。

Slice 3：Tests/docs

加入 fixture tests 與 README 用法。驗收：pass、blocked、proceed_with_assumptions 都有覆蓋。

Verification plan 驗證計畫

  • Incomplete artifacts 應讓 pre-spec gate 回傳 blocked 與 non-zero exit。
  • 完整 spec kernel 應讓 gate 回傳 pass。
  • 未承認風險時不得回傳 proceed_with_assumptions。
  • write-spec 在 gate pass 前必須失敗。
  • 完成 OpenSpec 後仍需刷新 source-manifest 並驗證 explainer。

Risk register 風險清單

辯論變成阻礙

緩解：要求 constructive disagreement、fast path、可接受 assumptions。

script 造成假信心

緩解：清楚分離 deterministic checks 與 agent judgment。

直接繞過 workflow

緩解：SKILL.md 將 pre-spec gate 定為 create/update normal path 的必要步驟。

proceed_with_assumptions 被濫用

緩解：必須有使用者明確承認，並寫入 proposal/design。

Rollout / rollback

先新增 helper 與文件規範，不移除既有 openspec-propose、manifest、explainer、archive scripts。若 workflow 造成過度摩擦，可回退為只在高風險或模糊 change 強制 gate，但此 change 的推薦方向是 normal create/update path 都需 gate。

Current status
